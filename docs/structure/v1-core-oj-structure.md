# V1 Structure - Core OJ

## 1. 文档信息

- 版本：V1
- 对应问题文档：`docs/problem/v1-core-oj.md`
- 对应需求文档：`docs/prd/v1-core-oj-prd.md`
- 文档类型：Structure Layer（定义系统结构，不定义技术实现）

## 2. 结构目标

本结构文档只回答一件事：**Core OJ 在 V1 阶段长什么样**。

V1 结构围绕 pure OJ 的最小闭环展开，使系统在抽象层面具备以下能力：

- 承载题目练习
- 承载代码提交
- 承载结果反馈
- 承载提交追踪
- 承载题目运营
- 承载判题运维

## 3. 系统结构总览

V1 Core OJ 由三类结构面组成：

1. 用户练习面：面向普通用户，完成做题闭环
2. 管理运营面：面向管理员，完成题目与用例维护
3. 判题执行面：承载提交处理、状态流转与结果回写

抽象结构如下：

```text
+--------------------------------------------------------------+
|                        Core OJ System                         |
|                                                              |
|  +------------------+      +-------------------------------+ |
|  | 用户练习面         |      | 管理运营面                      | |
|  | - 题目列表         |      | - 题目管理                      | |
|  | - 题目详情         |      | - 题目编辑                      | |
|  | - 提交结果         |      | - 用例管理                      | |
|  | - 提交记录         |      | - 提交监控                      | |
|  +------------------+      +-------------------------------+ |
|                 \                    /                        |
|                  \                  /                         |
|                   +----------------+                          |
|                   | 判题执行面        |                          |
|                   | - 提交接收        |                          |
|                   | - 状态流转        |                          |
|                   | - 结果产出        |                          |
|                   +----------------+                          |
+--------------------------------------------------------------+
```

## 4. 角色结构

V1 只有两类核心角色：

- 普通用户：做题、提交、查看自己的结果与记录
- 管理员：维护题目、维护测试用例、查看全局提交与异常状态

角色与结构面的对应关系如下：

| 角色 | 进入面 | 主要目标 |
|---|---|---|
| 普通用户 | 用户练习面 | 完成看题、提交、回看、继续练习 |
| 管理员 | 管理运营面 | 保证题目可用、判题可用、问题可定位 |

## 5. 用户练习面结构

用户练习面由四个核心页面组成：

### 5.1 题目列表

作用：

- 提供题目入口集合
- 提供搜索、筛选与状态感知
- 帮助用户决定下一题做什么

核心信息块：

- 题号 / 标题
- 难度
- 用户状态（未尝试 / 未通过 / 已通过）
- 进入题目详情的入口

### 5.2 题目详情

作用：

- 承载单题的完整练习空间
- 让用户在同一页面理解题目并进行提交

核心信息块：

- 题面内容
- 输入输出说明
- 样例
- 约束说明
- 可选语言
- 代码编辑区
- 提交入口

### 5.3 提交结果

作用：

- 呈现单次提交的反馈
- 帮助用户判断下一步动作

核心信息块：

- 判题结果
- 编译或运行摘要
- 通过情况摘要
- 返回题目 / 再次提交 / 下一题入口

### 5.4 提交记录

作用：

- 承载用户对历史尝试的回看能力
- 将练习过程从“单次动作”变成“连续轨迹”

核心信息块：

- 提交时间
- 语言
- 状态
- 进入某次提交详情的入口

## 6. 管理运营面结构

管理运营面由四个核心模块组成：

### 6.1 题目管理

作用：

- 查看题目列表
- 搜索与过滤题目
- 进入新建、编辑、发布、下线动作

核心信息块：

- 题目编号
- 标题
- 当前状态
- 更新时间
- 编辑与发布动作

### 6.2 题目编辑

作用：

- 编辑题目基础信息与题面内容
- 组织题目对用户侧的展示形态

核心信息块：

- 标题
- 难度
- 可用语言
- 题面主体
- 样例与约束入口
- 保存草稿 / 发布入口

### 6.3 用例管理

作用：

- 维护题目判题依据
- 区分样例与隐藏用例
- 控制用例是否参与实际判题

核心信息块：

- 用例编号
- 用例状态
- 是否隐藏
- 新增 / 删除 / 启停动作

### 6.4 提交监控

作用：

- 让管理员从全局视角查看提交状态
- 对异常提交和系统问题进行定位

核心信息块：

- 提交列表
- 状态过滤
- 失败分类
- 题目维度与语言维度入口
- 重试或处置入口

## 7. 判题执行面结构

判题执行面不直接服务某个页面，而是承担系统的执行主链路。

其结构可抽象为三个连续模块：

### 7.1 提交接收模块

作用：

- 接收来自用户练习面或程序化调用的提交请求
- 为每次提交生成可追踪实体

### 7.2 状态流转模块

作用：

- 维护提交从“进入系统”到“产出结果”的过程状态
- 让用户侧和管理侧都能读取统一状态

### 7.3 结果产出模块

作用：

- 输出最终判题结论
- 输出可回放的单次结果
- 为提交记录与提交详情提供事实来源

## 8. 核心对象结构

V1 结构层关注对象之间的关系，不展开具体字段设计。

核心对象如下：

| 对象 | 作用 | 与谁关联 |
|---|---|---|
| 用户 | 练习行为主体 | 与提交记录关联 |
| 题目 | 练习承载单元 | 与用例、提交、语言约束关联 |
| 测试用例 | 判题依据 | 归属于题目 |
| 提交 | 单次尝试记录 | 关联用户、题目、语言 |
| 判题结果 | 单次提交的反馈结果 | 归属于提交 |

对象关系如下：

```text
用户 -> 提交 -> 判题结果
题目 -> 测试用例
题目 -> 提交
题目 -> 可用语言
```

## 9. 核心流转结构

### 9.1 用户练习主链路

```text
题目列表
  -> 题目详情
  -> 发起提交
  -> 提交结果
  -> 再次提交 / 下一题 / 查看历史记录
```

### 9.2 管理运营主链路

```text
题目管理
  -> 新建或编辑题目
  -> 维护测试用例
  -> 发布题目
  -> 观察用户提交与判题状态
```

### 9.3 判题执行主链路

```text
接收提交
  -> 进入待处理状态
  -> 进入判题状态
  -> 产出完成结果或失败结果
  -> 回写给用户面与管理面
```

## 10. 状态结构

### 10.1 题目状态

V1 中题目至少存在以下结构状态：

- 草稿
- 已发布
- 已下线
- 会话私有

结构含义：

- 草稿：仅用于编辑，不对普通用户开放
- 已发布：进入公开练习面
- 已下线：从公开练习面移除
- 会话私有：仅供特定会话或程序化创建后的私域使用

### 10.2 提交状态

V1 中提交至少存在以下结构状态：

- 待处理
- 判题中
- 已完成
- 系统失败

这些状态同时服务于：

- 用户理解当前提交走到哪里
- 管理员定位执行面是否异常

## 11. 跨层结构约束

为保证后续 V2 可以复用 V1，结构层需要遵守以下约束：

### 11.1 写入能力不能只存在于管理端

题目、测试用例、提交等写入动作在结构上必须同时支持：

- 人工操作入口
- 程序化调用入口

### 11.2 用户面与管理面读取同一事实源

用户看到的提交状态和管理员看到的提交状态必须来自同一套提交事实，而不能出现两套结构定义。

### 11.3 结果必须可回看

提交结果不能只是瞬时反馈，而必须沉淀为结构化记录，供：

- 当前用户回看
- 管理员定位
- 后续 AI 编排层读取

## 12. 结构验收口径

当以下问题都能回答清楚时，说明 V1 结构完整：

- 用户从哪里进入题库、进入题目、进入结果、进入历史记录
- 管理员从哪里创建题目、维护用例、发布题目、查看异常
- 提交从哪里进入系统、如何流转、如何回到页面
- 题目、用户、提交、结果之间的关系是否单一且清晰
- 后续 AI 是否可以绕过 UI，直接复用同一套核心对象与流转
