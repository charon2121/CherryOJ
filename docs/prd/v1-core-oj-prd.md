# V1 PRD - Core OJ 闭环

## 1. 文档信息

- 版本：V1
- 对应问题文档：`docs/problem/v1-core-oj.md`
- 文档类型：Requirement Layer（定义做什么，不定义技术实现）

## 2. 产品目标

围绕 Online Judge 核心执行能力，构建最小可用的编程学习闭环，使用户可以在同一系统内完成：

学习内容获取 -> 练习编码 -> 提交运行 -> 结果反馈 -> 下一步引导

本 PRD 覆盖的交付范围为 **pure OJ**（题目、提交、判题、记录与基础流转）。

V1 的交付不仅包含用户侧做题体验，也包含让系统可运营所必需的：用户身份、题目管理、用例管理与判题处理能力。

## 3. 角色与权限（V1）

- 普通用户（User）：浏览题目、提交代码、查看自己的提交记录与结果
- 管理员（Admin）：创建/编辑/发布题目与测试用例，查看全局提交与判题状态

权限边界（V1）：

- 题目与用例的创建/修改/发布仅管理员可操作
- 普通用户不可直接访问题目的隐藏测试用例内容

## 4. 目标用户

- 编程初学者：需要从知识到练习的连续路径
- 进阶练习者：需要稳定的提交与判题反馈来迭代能力
- 管理员/运营者：需要通过管理端持续维护题目与用例，保障判题可用与可观测

## 5. 核心用户场景

### 场景 A：从学习到验证

1. 用户阅读某个知识点或题目说明
2. 用户在系统内完成代码编写并提交
3. 系统返回判题结果与关键反馈
4. 用户据此修改代码并再次提交

预期结果：用户无需切换外部平台即可完成最小学习验证闭环。

### 场景 B：基于结果继续学习

1. 用户完成一次或多次提交
2. 系统可读取并展示提交历史与结果状态
3. 用户根据结果进入下一题或下一学习步骤

预期结果：系统不仅提供“判对/判错”，还承载下一步训练入口。

### 场景 C：管理员创建并发布题目

1. 管理员在管理端创建题目草稿，填写题面信息
2. 管理员上传/维护测试用例（含隐藏用例）
3. 管理员发布题目，使其对普通用户可见

预期结果：系统可以持续新增题目并让用户侧可练习、可判题。

### 场景 D：判题失败的定位与恢复

1. 管理员看到提交失败/判题异常（例如编译器错误、系统错误）
2. 管理员能查看失败原因的基本信息与受影响范围
3. 管理员采取操作（例如下线题目、临时关闭语言、重试提交）

预期结果：判题系统具备基本可运维能力，避免“用户只能看到失败但无人可处理”。

## 6. 功能需求（V1）

### FR-1 题目与练习承载

- 系统应提供题目列表与题目详情的基础访问能力
- 每道题至少包含：题意、输入输出说明、约束/说明、样例
- 用户可明确知道当前要解决的问题和提交要求

### FR-2 代码提交

- 用户可针对指定题目进行代码提交
- 每次提交应绑定：题目、语言、代码内容、提交时间
- 用户可发起重复提交以进行迭代

### FR-3 判题反馈

- 系统应对提交给出可读的判题结果
- 判题结果至少覆盖：通过/未通过、基础状态信息、反馈时间
- 用户可基于结果判断是否继续修改

### FR-4 提交记录与结果追踪

- 用户可查看当前题目的历史提交记录
- 用户可区分不同提交的结果状态
- 用户可从记录中定位最近一次有效结果

### FR-5 学习闭环引导

- 在题目完成或提交结束后，系统应提供下一步入口（例如继续本题、进入下一题）
- 引导重点是让用户持续留在系统内完成学习流程，而非中断在单次判题

### FR-6 用户系统（身份与会话）

- 系统应支持用户注册与登录（V1 允许采用最简形式，但必须可区分用户身份）
- 系统应支持用户退出登录
- 提交记录应可按用户维度查询（用户至少可查看自己的提交）

### FR-7 管理端 - 题目管理

- 管理员可创建题目草稿、编辑题面内容并保存
- 管理员可设置题目状态：草稿 / 已发布 / 已下线
- 已发布题目对普通用户可见；草稿与下线题目默认对普通用户不可见

### FR-8 管理端 - 测试用例管理

- 管理员可为题目新增/删除/启用/禁用测试用例
- 测试用例支持标记为“隐藏用例”（普通用户不可见其内容）
- 题目发布前应具备至少一组可用于判题的用例集合

### FR-9 判题系统 - 提交生命周期

- 系统应为每次提交维护可追踪的状态流转（例如：排队中 -> 判题中 -> 已完成/失败）
- 系统应能产出稳定且可回放的判题结果，用于用户查看与管理端排障
- 系统应支持基本的重试/重新判题入口（至少管理员可触发）

### FR-10 管理端 - 提交监控与判题可运维

- 管理员可查看全局提交列表与过滤（按题目、语言、状态、时间）
- 管理员可查看判题失败的基本原因分类（例如：编译错误/运行错误/超时/系统错误）
- 当出现系统级异常时，管理端应提供明确的状态提示与基本处置入口（例如临时下线题目）

### FR-11 平台 API 能力（为 V2 奠基）

V1 的核心操作必须以稳定 API 暴露，不能仅依赖管理端 UI。具体要求：

- **题目创建与更新**应通过 API 完成（POST /api/problems），支持程序化调用，不只是表单提交
- **测试用例写入**应通过 API 完成，支持一次性批量创建（POST /api/problems/{id}/test-cases）
- **提交结果查询**应通过 API 返回完整判题明细（GET /api/submissions/{id}），包含各测试点结果
- **题目状态**除草稿/已发布/已下线三值外，应支持"会话私有（session-private）"状态，供 V2 AI 生成的题目使用，该状态下题目不出现在公开题库

> 该需求不影响 V1 用户侧体验，但决定 V2 AI 编排层能否落地，需在 V1 阶段一并完成。

## 7. 业务规则与边界

### 6.1 范围内

- 单用户基础学习闭环
- 题目 -> 提交 -> 判题 -> 继续练习的主链路
- 用户身份与提交记录绑定
- 管理端可维护题目与测试用例并控制题目上架
- 判题状态可追溯，支持最小可运维动作

### 6.2 范围外（V1 非目标）

- 复杂竞赛系统与实时比赛
- 多用户排名、对战、社区协作
- 覆盖所有编程语言与学习方向
- 复杂的题目权限体系（班级/课程/付费/组织）
- 高级反作弊与代码相似度检测
- 学习路径编排与对话式学习引导（V2 引入）

## 8. 成功指标（验收口径）

### 7.1 功能验收

- 用户可在系统内完成“查看题目 -> 提交代码 -> 获取判题结果”的完整流程
- 用户可查看同一题目的多次提交结果
- 系统在一次提交结束后提供明确下一步入口

### 7.2 功能验收（管理端与运营）

- 管理员可创建题目、维护用例并发布，使题目在用户端可见且可判题
- 管理员可查看全局提交与基本失败原因，用于定位判题异常
- 系统级判题异常不会表现为“完全不可解释的失败”（至少有可读的失败分类与时间信息）

### 7.2 体验验收

- 关键路径不依赖外部平台跳转
- 用户能够理解当前状态（题目状态、提交状态、结果状态）

### 7.3 演进验收

- V1 的需求结构可直接扩展到后续版本
- 核心流程定义稳定，不因后续能力扩展而被推翻

## 9. 版本切分建议

- V1.0：完成核心 OJ 闭环（题目、提交、判题、记录、下一步入口）
- V1.x：增强反馈表达与学习引导质量，不扩大到复杂竞赛与协作体系

## 10. 依赖与假设

- 假设系统具备最小可用的题目与判题能力
- 假设用户在单设备、单会话内完成主要学习流程
- 假设后续版本会在本 PRD 基础上继续扩展功能边界
