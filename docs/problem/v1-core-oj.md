# V1 Problem - Core OJ

## 背景

本项目希望最终把“学习、练习、提交、判题、反馈、下一步引导”整合成连续闭环（总体愿景见 `docs/problem/v0-vision.md`）。

要实现这件事，系统首先必须具备稳定的执行底座：题目可承载、代码可提交、结果可验证、状态可追溯。否则无论学习内容如何组织，都缺少可验证的训练闭环。

在学习编程语言、算法和软件开发的过程中，用户通常需要多个割裂的工具：

- 教程平台：用于获取知识
- Online Judge：用于提交代码和验证结果
- 任务管理或学习计划工具：用于组织学习路径

这些工具之间通常缺少统一的执行与反馈闭环。用户需要自己在“学习、练习、提交、分析、继续下一步”之间频繁切换，导致学习过程割裂、反馈滞后、路径不清晰。

## 核心问题

1. 编程学习过程缺少统一的平台来承载“教学 + 练习 + 自动验证 + 后续引导”
2. 现有 OJ 平台通常只负责判题，不负责教学编排和个性化学习路径
3. 用户在“看题、写代码、提交、查看结果、继续练习”之间切换成本高，难以形成连续训练节奏
4. 用户很难基于历史提交结果持续推进练习路径

## V1 要解决什么（范围）

V1 聚焦做一个 **pure OJ**，把最关键的执行闭环跑通并可追溯：

- 题目可浏览（列表/详情）
- 代码可提交（按题目与语言提交）
- 提交可判题（返回明确结果）
- 结果可回看（提交记录与状态追踪）
- 练习可继续（再次提交/进入下一题的明确入口）

V1 交付的价值在于把“可执行、可验证、可追溯”的底座打牢，为后续更高层的学习编排提供事实依据。

构建一个以 Online Judge 为核心执行引擎的系统，使用户可以：

- 在单一系统中完成题目练习、代码提交与结果回看
- 通过真实判题结果驱动后续练习与迭代
- 逐步从基础题练习扩展到更复杂训练任务

## 核心价值

让练习从“看题到验证”形成连续动作，并把每次尝试沉淀为可回看的提交历史，从而支持持续迭代与下一步练习。

## 非目标

- 初期不追求完整替代大型商业 OJ 平台
- 初期不支持复杂竞赛系统、多用户排名、实时比赛
- V1 不引入学习路径编排与对话式引导能力（该类能力在后续版本引入，见 `docs/problem/v2-ai-playground.md`）
- 初期不追求覆盖所有编程语言和所有学习方向

## 成功标准

- 用户可以在一个系统中完成”看题 -> 写代码 -> 提交 -> 获取结果 -> 再提交/下一题”的基础闭环
- 用户可以查看同题多次提交的结果与状态，并以此进行迭代
- 系统具备可扩展性：在不破坏 V1 主链路的前提下，后续可接入更多学习模式

---

## V1 为 V2 奠定的平台契约

V1 不只是一个可供用户直接使用的 OJ，它同时是 V2 AI 编排层的**执行基础设施**。

V1 需要确保以下能力以 API 形式稳定可用，V2 的 AI 将直接调用这些接口完成动作：

| 能力 | V2 用途 |
|---|---|
| 题目创建 API（含测试用例） | AI 在系统内生成练习题 |
| 提交记录查询 API（按提交 ID） | AI 读取真实判题结果，基于此继续对话 |
| 题目详情查询 API | AI 向用户展示它生成的题目内容 |
| 用户身份识别 | AI 会话与用户身份关联，提交记录可归属 |

> 设计原则：V1 不能假设题目只由人工在管理端创建。所有内容写入操作都应通过 API 而非仅通过管理端 UI 完成，以支持 V2 的程序化调用。

后续版本规划见 `docs/problem/v2-ai-playground.md`。
