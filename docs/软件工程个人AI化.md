# 软件工程个人 AI 化

关键不是把传统流程搬过来，而是要搞清楚：

> **传统流程里哪些是"本质能力"，哪些只是"组织分工"**

---

# 一、先把旧流程"拆解本质"

传统流程：

```text
PM → 原型 → UI → 开发 → 测试 → 上线
```

其实混了两种东西：

---

## 本质能力（必须保留）

这些是**永远不会消失的**：

1. **需求澄清（Problem Definition）**
2. **结构设计（System Design）**
3. **交互设计（Interaction Design）**
4. **实现（Implementation）**
5. **验证（Validation）**
6. **迭代（Iteration）**

---

## 组织分工（可以弱化）

这些只是"人类团队分角色"，分工本质上是沟通成本的产物：

- PM
- UI 设计师
- 前端
- 后端
- QA

AI 时代发生的变化是：

> 分工弱化，能力保留，被你一个人掌控

---

# 二、AI 时代的核心流程

你要构建的不是"新流程"，而是：

> **把能力重新组合成一个"单人可控闭环"**

---

## 核心流程

```text
1. 定义问题（Why）
2. 抽象结构（What）
3. 设计系统（How）
4. 生成实现（Build with AI）
5. 验证与修正（Validate & Fix）
6. 迭代演进（Evolve）
```

---

## 和传统流程对比

| 传统      | AI 时代        |
| --------- | -------------- |
| PM 写需求 | 你定义问题     |
| 原型 / UI | 你定义结构     |
| 开发      | AI 生成代码    |
| 测试      | 你主导设计验证 |

---

## 重要：这个流程不是线性的

很多人看到这个流程，会以为它是从上到下一次跑完的。**不是。**

实现会反向暴露规格的问题：

```text
定义问题
   ↓
 需求 / 结构 / 设计     ← ←（实现发现规格有误时，向上修正）
   ↓                          ↑
 AI 生成实现    →   →   →   →  ↑
   ↓
 验证
   ↓
 迭代
```

写代码发现 API 设计有缺陷、跑起来发现性能假设不成立、测试发现需求有歧义——这不是流程的失败，而是**正常的**。

> 规格被实现修正，是健康的迭代信号，不是需要避免的事。

每一次"向上修正"都应该落回文档，让规格和代码始终保持一致。

---

# 三、核心因果链

```text
问题（Problem）
   ↓
需求（Requirement）
   ↓
结构（Structure）
   ↓
系统（System）
   ↓
实现（Code）
```

如果跳过其中任何一层：

- **没需求** → 做出来没用
- **没结构** → 系统混乱
- **没设计** → 代码不可维护

---

# 四、层级定义

## 层 1：Problem（问题定义）

```text
用户有什么问题？
为什么要做？
```

## 层 2：PRD（需求）

```text
系统要提供什么能力？
边界是什么？
```

## 层 3：Structure（结构 / 原型）

```text
有哪些模块？
页面怎么流转？
```

## 层 4：Design（技术设计）

```text
API、数据结构、架构
```

## 层 5：Build（AI 实现）

```text
生成代码
```

## 层 6：Validate（验证）

```text
你主导设计测试；AI 辅助执行
```

---

**第 6 层需要单独展开。**

验证是整个流程中最难被压缩的环节。AI 生成的代码通常可以运行，但：

- **边界情况**：AI 不会主动想到你没提到的边界
- **安全漏洞**：AI 可能生成可用但不安全的代码
- **集成问题**：单模块测试通过，不代表整体流程没问题
- **盲点共享**：如果你用 AI 写代码，再用 AI 写测试，它们会共享相同的盲点

验证的核心是：**你要主动设计测试用例，而不是让 AI 自我检验。**

尤其要关注：
1. 正常路径之外的边界情况（空输入、超大值、并发）
2. 你规格里没有明确说明的隐含假设
3. 集成路径（模块之间的衔接，而不只是单元测试）

> 验证是"你的工作"，AI 只是执行工具。

---

# 五、AI 不会替你做"抽象"

AI 很强在：

- 写代码
- 生成 UI
- 填细节

但它很弱在：

- 定义问题
- 划分边界
- 做取舍

所以：

> **你必须掌控"Problem → Structure"这一段**

---

## 这三步比看起来难得多

很多人以为，知道了"Problem → PRD → Structure"的框架，就会做这三步了。

不是的。

写一个好的 Problem 文档，需要你真的理解用户的核心痛点，而不是列需求清单。写一个边界清晰的 PRD，需要你能区分"必须做"和"想做"，并且能明确说出"不做什么"。写一个合理的 Structure，需要你能预判模块的分工和演进方向。

这三步是需要长时间刻意练习的判断力，不是套模板就能做好的事。

> **前三步不是"填空题"，是需要持续打磨的思维能力。**

每次做完一个功能，回头看自己写的 Problem 和 PRD，问：它有没有边界不清的地方？有没有把"想做的"混进了"要做的"？有没有遗漏了核心场景？这就是在训练这个能力。

---

# 六、AI 上下文的实际限制

AI 生成实现有一个被低估的实际约束：**上下文窗口**。

项目规模小的时候，你可以把整个规格 + 代码库一起给 AI，它能生成连贯的实现。但项目变大之后，AI 很难同时理解整个规格和整个代码库，会出现前后矛盾、遗漏约束、生成重复代码等问题。

这意味着：

- **Structure 层的模块切割非常关键。** 切得好，每个任务可以独立给 AI 处理，不需要 AI 同时理解整个系统。
- **每次给 AI 的任务要有明确的边界。** 不是"帮我实现这个功能"，而是"在 X 模块里，实现 Y 接口，依赖 Z，不需要处理 W"。
- **规格文档是 AI 的"外部记忆"。** 规格写得越清楚，AI 就能在更小的上下文里做出更一致的实现。

> 任务颗粒度是你控制 AI 质量的核心杠杆。

---

# 七、怎么落地

每做一个功能，强制走这 5 步：

### 1. 写一句问题（Problem）

```text
用户容易忘记任务
```

### 2. 写需求（PRD）

```text
用户可以设置提醒
边界：只做单次提醒，不做周期提醒
```

### 3. 写结构（Structure）

```text
Task / Reminder / Notification
Reminder 属于 Task，Notification 是独立发送模块
```

### 4. 写设计（Design）

```text
API + DB + Flow
POST /api/reminders  →  Reminder 表  →  触发 Notification
```

### 5. 让 AI 实现，自己主导验证

```text
AI 生成实现 → 你检查边界用例 → 发现问题 → 修正规格或实现 → 继续
```

**前四步你必须自己做，第五步 AI 是执行者，你是验证者。**

---

# 八、Spec Driven Development（规格驱动开发）

更高阶的形态：

```text
Spec（Problem + PRD + Structure + Design）
   ↓
AI
   ↓
System
```

代码只是"规格的编译结果"。

这个模型的前提是：规格足够清晰，AI 才能生成可信赖的实现。规格模糊，AI 只会生成"看起来能跑"的代码，而不是"解决了对的问题"的代码。

---

# 九、角色转变

你的角色是：

> **System Designer + AI Orchestrator**

不是：写 CRUD 的人，也不是让 AI 自由发挥的人。

---

# 十、一句话总结

> **AI 时代不是简化流程，而是"压缩角色，但保留思考层级"。**

你压缩的是分工，你保留的是判断力。
