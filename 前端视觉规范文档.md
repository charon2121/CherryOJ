# CherryOJ 前端视觉规范文档

## 1. 文档定位

本文档不是给设计师讨论灵感用的，而是给 AI 与工程实现直接执行的前端视觉规范。

目标：

- 统一 `cherry-web` 的全站视觉语言
- 让用户端与管理端保持同一产品家族感
- 避免页面一页一个风格
- 避免 AI 常见的“紫色渐变 + 大圆角 + 营销页感”审美

适用范围：

- 首页
- 题库页
- 题目详情页
- 登录 / 注册页
- 管理后台
- 所有新建页面与组件

---

## 2. 总体风格

CherryOJ 的视觉风格定义为：

**现代极简、冷静专业、信息密度适中偏高、以工具感为主，带少量 Cherry 品牌强调。**

如果要类比，可参考：

- 用户端：`Linear + 少量竞赛平台工具感`
- 管理端：`Linear / GitHub / Notion Admin` 的专业后台秩序

### 2.1 风格关键词

- 克制
- 清晰
- 精确
- 专业
- 高密度但不压迫
- 少装饰，多秩序
- 产品感强于展示感

### 2.2 禁止风格

严禁出现以下风格：

- 大面积紫色 / 蓝紫色渐变
- 过度 Dribbble 化装饰
- 营销页式夸张 hero
- 高饱和荧光配色
- 超大圆角、超重阴影、玻璃拟态泛滥
- 无意义的漂浮卡片
- 为了“设计感”打碎信息秩序
- 一页里混用多套视觉语言

---

## 3. 色彩系统

## 3.1 基础原则

全站颜色只允许三层：

1. 中性灰阶：作为背景、边框、正文、弱文本的主体
2. Cherry 强调色：作为品牌主强调
3. 功能状态色：成功 / 警告 / 危险

除这三层之外，不应再引入额外情绪色作为主视觉。

## 3.2 主题基线

当前项目已接入 HeroUI 主题令牌，必须优先使用以下语义 token：

- `--background`
- `--foreground`
- `--surface`
- `--surface-secondary`
- `--surface-tertiary`
- `--border`
- `--muted`
- `--accent`
- `--success`
- `--warning`
- `--danger`

不要绕过 token 体系直接到处手写随机颜色。

## 3.3 品牌强调色

CherryOJ 的唯一主强调色为：

- `accent / cherry red`

使用范围：

- 主按钮
- 当前选中项
- 关键链接 hover
- 重要状态点缀
- 首页标题或关键数字的少量强调

禁止：

- 整页大面积铺满红色
- 同时再引入第二主色

## 3.4 灰阶策略

全站优先使用 `zinc / neutral` 风格灰阶。

使用建议：

- 页面背景：`background`
- 卡片背景：`surface`
- 次级区块：`surface-secondary`
- 分隔线：`border`
- 主文本：`foreground`
- 弱文本：`muted`

不要在同一页面里混用多种灰系逻辑。

## 4. 字体与排版

## 4.1 字体原则

当前项目字体已确定：

- 正文字体：`DM Sans`
- 等宽字体：`JetBrains Mono`

必须继续沿用，不要再切回 `Inter / system-ui / Roboto / Arial`。

## 4.2 标题规则

- 标题字重偏中高，但不过度粗黑
- 标题追求精确秩序，不追求夸张视觉冲击
- 首页主标题可以更大，但必须收敛，不得做广告横幅式夸张版式

## 4.3 正文规则

- 正文以 14px 到 16px 为主
- 行距适中偏松，确保信息密度高但不拥挤
- 弱说明文本统一降权，不与正文抢层级

## 4.4 等宽字体使用

等宽字体只用于：

- 题号
- 提交 ID
- 代码
- 时间 / 内存等数据指标

不要把等宽字体用于普通正文标题。

---

## 5. 间距与密度

## 5.1 总体密度

CherryOJ 页面密度定义：

- 用户端：中密度
- 管理端：中高密度

页面不能太松，也不能像传统后台那样拥挤。

## 5.2 间距原则

- 大区块之间：明显分层
- 卡片内部：紧凑但可呼吸
- 表格行高：偏紧凑
- 操作按钮区：保持克制，不做大按钮堆叠

## 5.3 圆角与阴影

- 圆角：小到中小
- 阴影：很轻
- 优先使用边框建立层次，而不是靠重阴影

目标是接近 Linear，而不是卡片型营销站。

---

## 6. 组件规范

## 6.0 组件优先级

组件选择必须遵守以下优先级：

1. 能直接用 HeroUI 组件完成，就不要回退原生元素
2. HeroUI 没有页面级成品组件时，优先用 HeroUI 基础组件组合实现
3. 只有组件库无法覆盖语义或交互时，才允许写原生元素或自定义基础组件

禁止：

- 已有 `Button / Input / TextArea / Dropdown / Link / Card / Table` 可用时，继续手写原生替代
- 一半用组件库、一半用原生元素，导致状态、密度和交互不统一

## 6.0.1 图标策略

前端统一图标策略如下：

- 优先引入成熟图标库，不再长期维持“页面里临时手写 SVG”
- 推荐图标库：`lucide-react`
- 新增图标应优先从统一图标库中选择，再做少量封装
- 只有品牌 Logo、业务专属图标、组件库缺失图标时，才允许自定义 SVG

禁止：

- 同一页面混用多套风格差异明显的图标
- 为普通后台操作反复手写临时 SVG

## 6.1 Button

按钮必须使用明确层级：

- `primary`：主操作，一屏 1 到 2 个
- `secondary`：次操作
- `tertiary / outline / ghost`：弱操作
- `danger / danger-soft`：删除、危险动作

禁止：

- 同一块区域放多个主按钮
- 所有按钮都用高强调色

## 6.2 Card

卡片只用于：

- 信息分组
- 数据块承载
- 功能区域切分

卡片不应用于：

- 为了“好看”给所有内容都套卡片
- 制造多层套娃

卡片视觉必须：

- 边框轻
- 背景干净
- 阴影极弱或无

## 6.3 Table

表格是 CherryOJ 的核心组件之一，必须具备：

- 强秩序
- 清晰列头
- 紧凑行高
- 弱 hover
- 清晰的数据对齐

题库页和管理后台优先使用表格，而不是卡片瀑布流。

## 6.4 Input / TextArea / Select

表单控件必须：

- 轮廓清晰
- 高度统一
- 标签层级稳定
- 聚焦反馈克制但明确

不要用大面积彩色描边，也不要做重阴影输入框。

## 6.5 Badge

Badge 只用于：

- 状态
- 难度
- 标签
- 环境信息

Badge 必须小、轻、清晰。
不要把 Badge 做成页面主视觉。

---

## 7. 页面模板规范

## 7.1 首页

首页不是营销官网，而是产品入口页。

必须包含：

- 冷静克制的产品价值说明
- 核心指标概览
- 题库 / 提交 /运行环境等关键能力展示
- 明确进入主流程的入口

首页应避免：

- 花哨 slogan 堆砌
- 空洞品牌口号
- 大量无意义装饰图形

## 7.2 题库页

题库页必须是：

- 表格型结构
- 搜索 / 筛选作为 client leaf
- 页面骨架作为 server shell

题库页不应做成：

- 大卡片列表
- 杂乱标签墙
- 瀑布流

## 7.3 题目详情页

题目页固定模板：

- 左：题面
- 右：编辑器与运行提交区

左侧强调阅读秩序，右侧强调操作效率。
二者必须显式分层，不能混成一个页面流。

## 7.4 后台列表页

后台列表页必须：

- 高密度
- 强表格
- 操作集中
- 说明文字克制

## 7.5 后台编辑页

后台编辑页应使用：

- server shell 承载页面框架
- client form 承载表单交互
- 双栏或主副栏结构

不允许把整页做成大块表单堆叠且毫无层次。

---

## 8. 动效与交互

## 8.1 动效原则

动效必须少而准。

允许：

- hover 颜色变化
- 菜单展开
- 页面局部渐入
- loading / pending 的最小反馈

禁止：

- 弹跳式浮夸动画
- 所有元素都动
- 重型过渡效果

## 8.2 交互反馈

交互反馈必须满足：

- hover 可见
- focus 可见
- active 可见
- disabled 明确

不可只依赖颜色变化微弱到不可感知。

---

## 9. 实现约束

## 9.1 技术约束

- 默认使用 Server Components
- 只有交互叶子组件使用 `"use client"`
- 页面结构优先 `server shell + client leaf`
- 主题必须基于 `globals.css + theme.css` 体系
- 优先使用 HeroUI v3 组件

## 9.2 代码约束

AI 在写页面时必须：

- 先判断页面属于用户端还是管理端
- 复用现有设计 token
- 复用已有 layout 边界
- 不引入第二套风格
- 不为局部页面单独发明一套审美

## 9.3 状态约束

不要把业务主数据放到全局客户端 store。

允许的 store 仅限：

- auth snapshot
- draft store
- pure UI state

---

## 10. 明确禁止事项

AI 实现前端时，以下行为直接视为不合格：

- 随手写紫色渐变主页
- 使用“白底 + 大圆角 + 大阴影 + 漂浮卡片”的模板页
- 把用户端做成营销页，把后台做成大屏
- 在同一页面混用 3 套以上按钮风格
- 在没有必要时把整个页面写成 Client Component
- 绕开 token 体系到处手写随机颜色
- 为了“设计感”牺牲信息密度和可用性

---

## 11. 本项目当前执行方向

CherryOJ 当前统一视觉方向为：

**现代极简、专业工具感、Linear 式秩序、Cherry 红作为唯一品牌强调。**

执行优先级：

1. 先统一用户端首页
2. 再统一题库页
3. 再统一题目详情页
4. 最后统一后台页面

后续所有页面重做和新增页面，都必须以本文档为唯一视觉基准。
